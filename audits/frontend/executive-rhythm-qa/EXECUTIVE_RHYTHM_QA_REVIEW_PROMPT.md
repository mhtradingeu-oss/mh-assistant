## Review dimensions

### 1. Authority

Identify the current frontend authority layers:

- App shell authority
- Router authority
- Shared context authority
- Page render authority
- AI panel authority
- Executive card authority
- Button/action authority
- CSS authority

Find whether there are duplicate competing authorities.

Report:
- canonical file
- competing file/block if any
- risk level
- recommended decision

### 2. Runtime safety

Check for risks in:

- route changes
- page refresh
- event listeners
- page mount/unmount
- global state mutation
- duplicated init flows
- overlays or loading locks
- project context
- AI workspace context
- data/projects accidental changes

Report anything that can cause:
- stuck loading
- double rendering
- broken page navigation
- stale context
- duplicate AI output
- visual flicker
- hidden click blockers

### 3. Executive cognition

For each priority page, answer:

- What is the page trying to help the user decide?
- Is the next best action clear?
- Is the AI role clear?
- Is the page showing too much raw data?
- Is there a clear difference between status, action, evidence, and output?
- Is the page executive-ready or still admin/debug-like?

Score each page from 1 to 10.

### 4. Cross-page consistency

Compare Home, AI Command, and Governance.

Check consistency of:

- page title area
- status cards
- action buttons
- AI CTA
- right panel
- card spacing
- typography
- badges
- empty states
- data/evidence sections
- button naming
- refresh/open workspace actions

Identify what should become the global pattern.

### 5. Visual design

Evaluate from a premium international SaaS standard.

Check:

- typography size
- readability
- density
- spacing
- card hierarchy
- color contrast
- sidebar balance
- topbar balance
- responsive behavior
- mobile/tablet risk
- visual calmness
- design repetition
- overloaded sections

Important:
The screenshots show that Home and AI Command are still too dense. Governance is cleaner but still flat in lower sections.

### 6. Final polish readiness

Identify:

- what is safe to keep
- what must be refactored
- what must be redesigned
- what must be deleted
- what must be consolidated
- what needs documentation
- what can become the final design system

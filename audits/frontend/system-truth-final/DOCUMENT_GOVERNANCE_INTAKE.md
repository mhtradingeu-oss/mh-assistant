# Document Governance Intake

## Purpose

This intake exists because MH-OS now contains many audit, planning, closeout, readiness, checkpoint, and migration documents.

The goal is not to delete documents immediately.

The goal is to classify them safely before future cleanup.

---

## Current Problem

The repository contains many documents across:

- audits/frontend
- audits/runtime-governance
- audits/release
- audits/checkpoints
- docs
- page-specific audit folders
- historical patch/closeout folders

Some documents are still active.
Some are canonical.
Some are historical evidence.
Some are superseded by newer decisions.
Some are duplicates or partial planning artifacts.
Some may eventually be archived.
Only a small number may later be deleted, and only after review.

---

## Governance Principle

Do not delete project knowledge before classification.

Archive before delete.

Prefer preserving historical evidence unless it is empty, generated by mistake, or explicitly superseded and duplicated elsewhere.

---

## Required Future Classification

Every document should eventually be classified as one of:

- Canonical
- Active
- Transitional
- Historical
- Superseded
- Duplicate
- Archive Candidate
- Delete Candidate

---

# Workflows Architecture Notes

## Current Reality

Workflows is already one of the strongest operational surfaces in MH-OS.

Implemented:
- workflow orchestration
- automation planning
- AI refinement bridges
- durable handoffs
- approvals
- auto mode
- pause/resume
- execution packages
- workflow intelligence
- task conversion
- campaign transitions
- governance-aware execution

## Main architectural issue

Execution runtime is hidden inside:
- render flow
- action buttons
- workflow forms

instead of being exposed as:
- execution operating system
- runtime coordination surface
- operator control room

## Existing strengths

Already production-grade:
- safe automation
- approval gates
- durable-first behavior
- orchestration lifecycle
- AI command bridge
- workflow intelligence loading
- recommendation integration

## Existing UX weakness

Current experience still feels like:
- workflow builder
- automation form
- execution helper

instead of:
- execution runtime
- operational command center
- orchestration room

## Canonical future role

Workflows should become:
- execution OS
- automation coordination layer
- approval runtime surface
- orchestration monitor
- operator control room

## Protected systems

Do NOT rewrite:
- automation-engine
- approval lifecycle
- durable handoffs
- auto mode lifecycle
- workflow intelligence
- AI refinement bridge

## Activation strategy

Do:
- improve runtime visibility
- improve execution hierarchy
- expose runtime states
- expose approvals
- expose orchestration progress
- expose automation status

Do NOT:
- redesign from scratch
- rewrite orchestration
- replace automation engine

## Priority

High

Reason:
This page is already close to a real execution operating surface.

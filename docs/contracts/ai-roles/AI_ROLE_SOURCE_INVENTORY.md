## Current authority conclusions

- Backend operational role membership, service-domain ownership, reviewers, handoffs and escalation are governed by `backbone.js:114-262` (BACKEND_OPERATIONAL_AUTHORITY, HIGH).
- HTTP route access is key/scope/risk based in `runtime/orchestrator-service/lib/security/route-permission-catalog.js:154-203`, not AI-role based (BACKEND_SECURITY_AUTHORITY, HIGH). Role RBAC is therefore not proven by that catalog.
- Frontend route visibility uses backend-projected `route_permissions` when available and otherwise literal fallback tables (`authority-projection.js:66-88`; `route-role-fallback.js:53-66`) (FRONTEND_PROJECTION/COMPATIBILITY, HIGH).
- AI provider identity is produced from unvalidated request specialist IDs and mode aliases in `ai-orchestrator.js:182-219,271-340` (BACKEND_DOMAIN_MAPPING, HIGH). Provider access itself remains controlled by separate provider gates; no role-specific provider grant was found.
- Memory/context filtering by a unified role contract is not proven. AI Command creates customer/sales scoped context branches (`ai-command.js:5399-5413`), while backend AI memory projection is generic (`authority-projection.js:61-64`). Classification: UNKNOWN_REQUIRES_PROOF.

## Role dimensions and security boundary

The following are separate dimensions and must never be inferred from one another: conceptual identity; raw/source identity and provenance; backend operational authority identity; backend security key/scope identity; frontend projection identity; frontend experience identity; display label; AI Command lane; provider prompt persona; legacy/contextual alias; coordination mode; system actor; reviewer and escalation identity; route visibility; page owner/support; backend execution and action permission; workflow/task ownership; handoff source/destination; tool visibility, tool use and tool execution; provider capability authorization; memory scope; and context scope. Current proof for a value in one dimension grants nothing in another.

`system_orchestrator` is a `SYSTEM_ACTOR`, not a selectable human/specialist member, and must be excluded from normal selectable lists unless a future actor contract explicitly projects it. `team` is a `COORDINATION_MODE`, not a role or authorization identity. `media` and `seo` are contextual prompt/capability lanes where proven, not global authority aliases. Browser/session context is not a durable operational handoff.

## Identity-class boundary

Future contracts must type identity independently as `HUMAN_USER`, `AI_SPECIALIST`, or `SYSTEM_ACTOR`. A human account/session, an AI prompt persona, and a system automation actor are not interchangeable principals. Authentication identity, backend authority identity, prompt identity and audit actor identity require separate provenance. This is a future contract recommendation, not a statement that these identity classes are implemented today.

## Registration and middleware order

Observed order is fragmented: selected protected-route guards are registered first (before parsing); public-alias telemetry; Helmet/CORS/compression and public-alias classifier; more protected guards; JSON parser; shared write guard; shared read guard; sensitive runtime security; limited rate limiting; project slug validation; handlers; global error handler; then additional media routes. There is no authentication or principal-resolution stage. Governance gates occur inside selected handlers. Provider classification occurs in the sensitive runtime middleware. Logs occur on selected denials/errors, not every allow/deny.

Routes registered after `runtime/orchestrator-service/server.js:23706-23739` still traverse earlier global middleware but their thrown/forwarded errors cannot reach that earlier error handler. Direct publish routes at 23942/23958 are handler-denied 403; the legacy publish polling bridge at 23978 is disabled 410.
